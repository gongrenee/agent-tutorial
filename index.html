CHAPTER 01 · CORE CONCEPT

🤖 什么是 AI Agent？

在写代码之前，先建立最重要的认知：Agent 是一种全新的软件形态，和你熟悉的网站、App 有本质不同。

AI Agent 在软件世界里是什么位置？

你可能用过网站、App、小程序——它们都是软件，但 AI Agent 和它们有本质区别：逻辑不是开发者写死的，而是 LLM 实时推理决定的。

THREE KEY DIFFERENCES

①
变量声明：Python 直接写，TS 要加 const / let
Python 不需要声明变量类型，直接 x = 1 就行。TS 必须先声明 const x = 1，更严格但也更安全。
②
异步处理：TS 需要 await，Python 默认同步
调用 API 需要等网络返回结果。Python 默认"等完再继续"（同步），TS 默认"边等边做其他事"（异步），所以必须写 await 告诉它"在这里等一下"。
③
符号习惯：TS 多了 ; { } new 等符号
TS 继承自 C 语言家族，语句结尾要加 ;，创建对象要加 new，参数对象用 { } 包裹。Python 去掉了大部分这类符号，读起来更接近英文。

工作原理（超简版）

LLM 是在海量文本上训练的神经网络。核心能力只有一件事：根据给定文字，预测下一个最合理的词。

LLM PROCESSING PIPELINE

👤 输入文字 — "今天天气怎么样？"
↓
🔤 Tokenizer — 文字切分为 Token（词片段）
↓
🧠 Transformer 神经网络 — 数千亿参数处理
↓
✉️ 输出 — "我无法查天气，但可以…"

关键概念：Token

LLM 不按"字"处理，而是按 Token（词片段）处理。

  • 英文："Hello world" ≈ 2 个 Token
  • 中文："你好世界" ≈ 4–6 个 Token
  • Token 越多，处理越慢、费用越高
Token = 费用和速度的关键。Claude 计费单位就是 Token，输入 + 输出 Token 总量决定 API 调用成本。

中文开发者注意：中文比英文更贵

Tokenizer 基于英文语料训练，中文没有天然词边界，汉字往往一字一 token，导致同样意思的中文比英文消耗多 2–3 倍 token。

内容文字数Token 数成本倍数
"You are a helpful assistant."30 字符≈ 6 tokens基准
"你是一个有用的助手。"9 个字≈ 11 tokens约 1.8×
实用省钱技巧：
  • System Prompt 用英文写——模型理解没问题，但 token 少得多
  • 工具的 description 用英文——每次调用都会发送，积少成多
  • 用户对话该用中文就用中文——用户体验优先，成本影响有限

System Prompt vs User Message

调用 LLM API 时，你发送的不是一条消息，而是一个结构化的输入包，里面有两类完全不同用途的内容。搞清楚这两类，是写 Agent 的基本功。

SYSTEM PROMPT — 开发者写，用户看不到

在 API 调用里用 system= 字段传入。每次对话都会生效，相当于给 LLM 预设好角色和行为规则。

👉 Agent 开发中，System Prompt 就是你"定义这个 Agent 是什么"的地方：它能做什么、不能做什么、用什么语气、调用哪些工具。
USER MESSAGE — 用户发，每次不同

在 messages=[{"role":"user",...}] 里传入。是运行时用户实际输入的内容，每次请求都可以不同。

👉 Agent 开发中，User Message 是用户当前这一轮的请求：可以是问题、指令、或者要处理的数据。

在 API 里，它们长这样：

api_structure.py
# ① System Prompt 单独定义，避免多行缩进混乱（真实项目常见写法）
SYSTEM_PROMPT = "你是一个天气查询助手。" \
               "只回答天气相关的问题，其他话题拒绝回答。" \
               "用简洁的中文回复，不超过 3 句话。"

# ② 调用 API，传入 system 和 messages 两个字段
client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,   # 必填：限制回复最多用多少 Token
    system=SYSTEM_PROMPT,
    messages=[
        {"role": "user", "content": "上海今天天气怎么样？"}
    ]
)
关键理解：System Prompt 决定 Agent 的性格和能力边界，User Message 是 Agent 每次实际处理的任务。两者分离，让你能用同一套 LLM 搭出行为完全不同的多个 Agent。

多轮对话：让 LLM 记住上下文

LLM 本身没有记忆，每次 API 调用都是独立的。要让它"记住"对话，需要把历史一起发过去：

multi_turn.py
messages = []  # 存储对话历史

def chat(user_input):
    messages.append({"role": "user", "content": user_input})
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=1024,
        messages=messages  # 每次都发送完整历史
    )
    reply = response.content[0].text
    messages.append({"role": "assistant", "content": reply})
    return reply

print(chat("我叫小明"))       # "你好，小明！..."
print(chat("我叫什么名字？"))  # "你叫小明。"（记住了！）
⚠️ 发送历史记录的两个关键限制
① 上下文窗口有上限
每个模型能接受的 Token 总量有限（Claude claude-opus-4-5 是 200K Token）。对话轮数越多，历史越长，超过上限后 API 会报错。

应对方法：只保留最近 N 轮，或把旧对话压缩成摘要。
② 每轮费用递增
你每次都把完整历史发过去，意味着第 10 轮的 API 调用要付前 9 轮对话的 Token 费用。对话越长，单次调用越贵。

应对方法：长会话场景要做历史裁剪，避免无限增长。

模型选择与 Agent 适配

选什么模型，直接决定 Agent 能不能正常工作——而不只是"快一点慢一点"的差别。市面上主流模型来自不同厂商，能力侧重各不相同。

影响 AGENT 的四个核心模型能力

🔧
工具调用可靠性
能否稳定选对工具、正确填写参数。弱模型会漏参数或调错工具，导致 Agent 循环崩溃。
📐
上下文窗口大小
决定 Agent 一次能"看到"多长的历史和工具结果，直接限制记忆系统的设计上限。
📋
指令遵循能力
System Prompt 写得再好，模型不遵守等于白写。弱模型常常忽略约束或格式要求。
🗺️
多步推理深度
复杂任务规划需要足够强的推理能力。弱模型面对多步骤时容易走弯路甚至死循环。

CHAPTER 04 · FUNCTION CALLING

🔧 工具调用（Function Calling）

这是 Agent 最核心的技术！让 AI 不只会"说"，还能真正"做"——调用函数、操控真实系统。

工作流程

TOOL USE PIPELINE

👤 用户："上海现在几度？"
↓
🧠 LLM 决策：需要调用 get_weather 工具
↓
⚡ 你的代码：执行 get_weather("上海")
↓
🧠 LLM 综合工具结果，生成最终回答
↓
✅ "上海现在 28°C，晴天"
所有 LLM 都是这样，不只是 Claude：任何大语言模型本质上只能输出文字。"调用工具"的意思是 LLM 在回复里写下"我想用这个工具、传这些参数"——真正执行函数、拿到结果的永远是你的代码。LLM 没有直接操控任何东西的能力，你的程序才是执行者。
PART 1 · 工具定义（给 LLM 看）

一份 JSON 格式的"菜单"，告诉 LLM 有哪些工具可以用、每个工具需要传什么参数。LLM 根据这份菜单决策，它只能看到这里。

PART 2 · 函数实现（你自己写，LLM 看不到）

当 LLM 说"我要调用这个工具"后，你的代码负责真正执行。这里才是实际发生的事情——查数据库、调接口、发邮件都在这里。

Part 1：工具定义（菜单）

这份 JSON 会通过 tools= 参数发送给 LLM，LLM 读完后才知道有哪些工具可以选择。注意三个字段的作用：

name
工具的唯一标识，LLM 用这个名字"呼叫"工具
description
工具的用途说明——写得越清楚，LLM 选对工具的概率越高
input_schema
需要哪些参数、每个参数是什么类型，LLM 按这个格式填写参数

def get_weather(city: str) -> dict:
    data = {
        "上海": {"温度": "28°C", "天气": "晴天", "湿度": "65%"},
        "北京": {"温度": "22°C", "天气": "多云", "湿度": "45%"},
    }
    return data.get(city, {"error": f"找不到 {city} 的数据"})

def send_email(to: str, subject: str, body: str) -> dict:
    print(f"📧 发送邮件到: {to} | 主题: {subject}")
    return {"status": "success"}

# 调度器：根据 LLM 给出的工具名，路由到对应的函数
def execute_tool(name: str, inputs: dict):
    if name == "get_weather": return get_weather(**inputs)
    if name == "send_email":   return send_email(**inputs)
    return {"error": f"未知工具: {name}"}
execute_tool 的作用：LLM 返回的工具调用指令只有名字和参数（字符串），execute_tool 负责把名字映射到真实函数并执行。这是你控制权的体现——你可以在这里加权限校验、日志记录、错误处理。

CHAPTER 05 · AGENT LOOP

🔄 Agent 循环：ReAct 框架

单次 LLM 调用只能"想一次"——复杂任务需要多步推理、中途使用工具、根据结果调整方向。Agent 循环解决的正是这个问题：让 LLM 反复思考和行动，直到任务完成。

ReAct = Reason + Act（推理 + 行动）

三个概念，同一件事的不同层级：
ReAct 是每一轮循环里 LLM 的思考方式 → Agent 循环（本章）是把 ReAct 用 while True 串起来的代码结构 → Loop Engineering（进阶章）是你用自然语言指令驾驭这整套机制，让 Agent 自主迭代。

理解了 ReAct，就理解了循环的"每一步在干什么"；理解了循环，才能设计出真正可控的 Loop Engineering 指令。
🧠
THOUGHT — 思考
LLM 分析当前状态，决定下一步做什么
"我需要先查天气，再判断是否推荐带伞"
↓
⚡
ACTION — 行动
调用工具，例如 get_weather("北京")，真实执行函数
↓
👁️
OBSERVATION — 观察
收集工具返回的结果，反馈给 LLM 继续思考
↓ 循环，直到任务完成

Agent 主循环代码

下面这段代码是整个 Agent 的骨架。每一段都对应 ReAct 循环里的一个环节：

agent_loop.py
import json
import anthropic

client = anthropic.Anthropic()

def run_agent(user_message: str):
    # ① 初始化消息列表，放入用户的第一条消息
    messages = [{"role": "user", "content": user_message}]
    print(f"\n👤 用户: {user_message}\n")

    while True:
        # ② 让 LLM "思考"：把当前所有消息发给 LLM，拿到它的决策
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=4096,
            tools=tools,      # 告诉 LLM 有哪些工具可以调用
            messages=messages  # 包含完整的对话历史 + 工具结果
        )

        # ③ LLM 决定调用工具（stop_reason = "tool_use"）
        if response.stop_reason == "tool_use":
            results = []
            for block in response.content:
                if block.type == "tool_use":
                    print(f"  🔧 {block.name}({block.input})")
                    # ④ 你的代码真正执行工具，拿到结果（OBSERVATION）
                    result = execute_tool(block.name, block.input)
                    print(f"  ↳  {result}\n")
                    results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,   # 对应 LLM 发起的那次调用
                        "content": json.dumps(result, ensure_ascii=False)
                    })
            # ⑤ 把 LLM 的决策和工具结果都追加进历史，下一轮 LLM 能看到
            messages.append({"role": "assistant", "content": response.content})
            messages.append({"role": "user", "content": results})
            # ↑ 然后回到 while True 顶部，开始下一轮"思考"

        else:
            # ⑥ LLM 认为任务完成（stop_reason = "end_turn"），输出最终答案，退出循环
            answer = response.content[0].text
            print(f"🤖 Agent: {answer}")
            return answer

run_agent("上海今天天气怎么样？")
每一轮循环发生的事
① 初始化消息 → ② LLM 思考 → ③ 需要工具？ YES → ④ 执行工具 → ⑤ 结果存入历史 → 回②
③ 不需要工具 → ⑥ 输出答案，退出

CHAPTER 06 · MEMORY SYSTEM

🗄️ Agent 的记忆系统

有用的 Agent 需要"记住"事情。了解四种记忆类型及其实现方式。

四种记忆类型

💬
工作记忆
当前对话上下文窗口，对话结束即消失
📋
临时记忆
任务执行中的中间结果，存在变量里
🗄️
长期记忆
存入数据库，可跨会话永久保存
📚
外部知识
RAG 检索增强，从文档库中搜索相关信息

短期 vs 长期对比

🧠 短期（上下文窗口）
  • 存在 messages 列表里
  • 对话结束即消失
  • 有 Token 长度限制
  • 成本随历史增长而上升
💾 长期（外部存储）
  • 存在文件/数据库/向量库
  • 跨会话永久保存
  • 通过工具读写
  • 可以存储海量信息

实现简单的长期记忆 策略 ③ · 选择性记忆

memory.py
import json, os

# ① 记忆文件路径：Agent 停止后数据依然保留在这个 JSON 文件里
MEMORY_FILE = "agent_memory.json"

# ② 读取记忆：从文件加载已有的 key-value 数据
#    如果文件不存在（第一次运行），返回空字典 {}
def load_memory() -> dict:
    if os.path.exists(MEMORY_FILE):
        with open(MEMORY_FILE, "r", encoding="utf-8") as f:
            return json.load(f)
    return {}

# ③ 保存记忆：把整个 memory 字典写回文件
#    ensure_ascii=False 让中文正常存储，indent=2 让文件易读
def save_memory(memory: dict):
    with open(MEMORY_FILE, "w", encoding="utf-8") as f:
        json.dump(memory, f, ensure_ascii=False, indent=2)

# ④ 存入一条记忆：先读出旧数据，加入新 key，再整体写回
#    LLM 调用这个工具时传入 key（名称）和 value（内容）
def remember(key: str, value: str):
    memory = load_memory()   # 读出已有记忆，不覆盖其他条目
    memory[key] = value        # 写入新条目，key 相同则更新
    save_memory(memory)
    return {"status": "success", "saved": f"{key} = {value}"}

# ⑤ 查询一条记忆：用 key 查找，找不到时返回提示而不是报错
def recall(key: str):
    return load_memory().get(key, f"未找到: '{key}'")
生产环境中，长期记忆通常存在 SQLite 或向量数据库（如 Chroma、Pinecone）中，支持语义搜索——找到"含义相近"的记忆。

上下文管理技巧

对话历史越积越长，Token 成本随之上升，甚至超出上下文窗口。以下三种策略是实际项目中最常用的应对方式：

策略 ① · 截断
只保留最近 N 条消息
最简单粗暴的方案：超出数量就直接丢掉最早的消息，只把最近 N 轮发给 LLM。
✅ 实现简单，成本可控
❌ 可能丢失重要的早期信息
messages = messages[-N*2:]
# 保留最近 N 轮（每轮 2 条）
策略 ② · 摘要压缩
用 LLM 总结旧对话
当消息过多时，先让 LLM 把早期对话压缩成一段摘要，再把摘要 + 最近 N 条发给 LLM。
✅ 保留了关键信息
❌ 多一次 LLM 调用，有延迟和费用
summary = llm("总结以下对话...")
messages = [{"role":"user","content":summary}] + recent
策略 ③ · 选择性记忆
主动提取并存储要点
Agent 每轮结束后主动判断"这条信息值得记住吗"，值得的写入长期记忆，不需要的丢弃。
✅ 上下文始终精简高效
❌ 需要设计"什么值得记"的判断逻辑
# 对应上面 remember(key: str, value: str)
remember("user_name", "小明")
remember("user_city", "上海")
# 不重要的信息直接丢弃，不写入文件

    data = {
        "上海": {"天气": "晴天", "温度": "28°C"},
        "北京": {"天气": "多云", "温度": "22°C"},
        "深圳": {"天气": "阵雨", "温度": "30°C"},
    }
    return data.get(city, {"error": f"找不到 {city} 的数据"})

def calculate(expression: str) -> dict:
    try:
        allowed = set("0123456789+-*/()., ")
        if not all(c in allowed for c in expression):
            return {"error": "不安全的表达式"}
        return {"result": eval(expression), "expression": expression}
    except Exception as e:
        return {"error": str(e)}

def get_current_time() -> dict:
    now = datetime.now()
    return {
        "datetime": now.strftime("%Y年%m月%d日 %H:%M:%S"),
        "weekday": ["周一","周二","周三","周四","周五","周六","周日"][now.weekday()]
    }

def execute_tool(name: str, inputs: dict):
    if name == "get_weather":      return get_weather(**inputs)
    if name == "calculate":        return calculate(**inputs)
    if name == "get_current_time": return get_current_time()
    return {"error": f"未知工具: {name}"}

# ── Agent 主循环 ──────────────────────────────
client = anthropic.Anthropic()

def run_agent(user_input: str, history: list) -> str:
    history.append({"role": "user", "content": user_input})

    while True:
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=4096,
            system="你是智能助手，可查天气、做计算、查时间。用中文回答，简洁友好。",
            tools=TOOLS,
            messages=history
        )
        if response.stop_reason == "tool_use":
            results = []
            for block in response.content:
                if block.type == "tool_use":
                    print(f"   ⚙️  {block.name}({block.input})")
                    result = execute_tool(block.name, block.input)
                    results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": json.dumps(result, ensure_ascii=False)
                    })
            history.append({"role": "assistant", "content": response.content})
            history.append({"role": "user", "content": results})
        else:
            answer = response.content[0].text
            history.append({"role": "assistant", "content": answer})
            return answer

⚡ Loop Engineering：让 Agent 自主迭代

学完本课程，你已经具备了设计 Loop Engineering 的全部底层知识。Loop Engineering 的核心是：你不写代码，你设计循环指令，让 Agent 自己写、自己测、自己改，直到任务完成。

模式 ① 单循环
一个 Agent 独立完成任务，自我修正直到通过验证。

适合：代码生成、文章写作、数据处理
模式 ② 嵌套循环
外层 Agent 拆解任务，内层 Agent 各自完成子任务并迭代。

适合：复杂项目、多步骤工作流
模式 ③ 带评审循环
一个 Agent 生成，另一个 Agent 审核，不通过则打回重做。

适合：高质量要求、需要交叉验证

四个关键设计要素

① 明确任务定义
告诉 Agent 要做什么、完成标准是什么。越清晰，Agent 偏离越少。
② 可验证的终止条件
循环必须有明确出口——测试全部通过、达到质量分、人工确认。没有终止条件 = 死循环。
③ 最大迭代次数上限
兜底保护。建议 10-20 轮，超出后强制停止并报告卡点，而不是默默消耗 Token。
④ 工具权限最小化
只给 Agent 完成任务必需的工具。权限越大，出错代价越高，Loop 越难控制。

实用 Loop 指令模板

代码开发 Loop
你是一个 Python 开发 Agent。任务：[具体任务描述]。
工具：write_file（写代码）、run_tests（执行测试）、read_error（读取错误信息）。
循环执行直到 run_tests() 返回全部通过，或达到 15 轮上限。
每轮打印：轮次 / 测试结果 / 本轮修改点。超出上限报告卡点。
内容创作 Loop（带评审）
Writer Agent：根据大纲生成文章初稿。
Reviewer Agent：按以下标准评分（0-10）：逻辑性、可读性、覆盖度。
低于 8 分则返回具体改进意见，Writer 修改后重新评审。
通过后输出最终版本，最多迭代 5 轮。
Loop Engineering 的本质就是你在课程里学的 Agent 循环——只是把执行者从代码变成了 AI 自己。你理解了 stop_reason、工具调用、上下文管理，就能设计出真正可控的 Loop，而不是祈祷 Agent 别跑偏。

CHAPTER 09 · EVALUATION

📊 Agent 质量评估

会构建只是开始，能判断好坏才是高手。掌握评估 Agent 的核心维度与方法，让你的 Agent 从"能跑"走向"可靠"。

为什么评估如此重要？

和普通软件不同，Agent 的行为是概率性的——同一个问题，不同运行可能产生不同结果。没有评估体系，你无法知道：

  • Agent 在新场景下是否还能正常工作
  • 一次修改是改进了还是引入了退化
  • 当前 Agent 的瓶颈在哪里（Prompt？工具？循环逻辑？）
常见误区："我测了几条，Agent 回答得不错" ≠ Agent 质量好。个案测试无法覆盖真实分布，系统化评估才是基准。

三大评估维度

🎯
任务完成质量
最终答案是否正确？目标是否真正达成？结果有没有幻觉？
⚡
效率与成本
用了几轮循环？消耗多少 Token？平均延迟多少秒？
🛡️
可靠性与安全
工具失败时是否优雅降级？边界输入下行为是否合理？

        used_tools = []
        answer = run_agent_tracked(case["input"], used_tools)  # 带工具追踪

                   case["expect_tool"] in used_tools)
        kw_ok   = all(kw in answer for kw in case["expect_keywords"])
        passed  = tool_ok and kw_ok

    print(f"\n📊 通过率: {score:.0f}% ({len(results)} 个用例)")
    return score

CHAPTER 10 · MULTI-AGENT

🤝 多 Agent 协作设计

单个 Agent 有能力上限——复杂任务需要多个 Agent 分工协作。理解协作模式与设计陷阱，才能构建可靠的多 Agent 系统。

为什么需要多 Agent？

单 Agent 在以下场景会遇到瓶颈：

  • 上下文窗口限制：任务涉及的信息量超过单次 Token 上限
  • 并行效率：多个子任务可以同时推进，无需串行等待
  • 专业化分工：搜索、写作、代码、审核各有专精，一个 Agent 难以兼顾
  • 相互校验：一个 Agent 生成，另一个审核，减少幻觉和错误
经验法则：如果你发现单个 Agent 的 System Prompt 越写越长、工具越加越多，通常是拆分为多 Agent 的信号。

两种核心编排模式

🎯 集中式编排（Orchestrator）
  • 一个主 Agent 接收任务、分解步骤
  • 将子任务分派给专职子 Agent
  • 汇总结果，生成最终输出
  • 优点：流程清晰、易于调试
  • 缺点：主 Agent 是单点瓶颈
🔗 去中心化（Pipeline）
  • Agent 之间按流水线依次传递
  • 上一个的输出是下一个的输入
  • 无需中央控制节点
  • 优点：扩展灵活、模块解耦
  • 缺点：错误传播难以中途干预

    # 主 Agent：理解任务，分解步骤
    plan = orchestrator_llm(user_task)
    # plan = {"steps": [{"agent": "search", "input": {...}}, ...]}

    results = {}
    for step in plan["steps"]:
        agent_name = step["agent"]
        agent_input = step["input"]

        # ✅ 每个子 Agent 只拿自己需要的输入 + 前序结果

三大设计陷阱

陷阱现象避免方式
循环调用 Agent A 调 B，B 又调回 A，无限循环 维护调用链，检测重复节点；设置最大深度限制
错误静默传播 子 Agent 失败但返回空结果，主 Agent 继续处理，最终输出错误 子 Agent 失败必须返回明确的 {"error": "..."}，主 Agent 检查每步结果
上下文爆炸 把所有历史传给每个子 Agent，Token 暴增，成本失控 严格隔离上下文，每个子 Agent 只接收完成本步骤必需的信息

何时拆分为多 Agent？

单次任务超过 10 步
需要并行执行子任务
需要相互校验输出质量
System Prompt 超过 500 字仍不够用
工具超过 8 个难以管理
不同子任务需要不同模型（强模型规划 + 弱模型执行）
